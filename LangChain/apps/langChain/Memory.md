# 三、记忆模式实战手册

下面的表格展示了几种常见的记忆模式，接下来我们将逐一拆解它们的工作机制与适用场景。

## 记忆模式对比表

| 记忆类型 | 工作原理 | 优点 | 缺点 | 适用场景 |
|---------|---------|------|------|---------|
| **缓冲记忆（Buffer）** | 全量保存所有历史对话 | ✅ 无损、简单、通用<br>✅ 所有模型都支持 | ❌ Token 消耗快<br>❌ 长对话会超上下文限制 | 短对话、调试、小规模应用 |
| **缓冲窗口记忆（Window）** | 只保留最近 K 轮对话 | ✅ Token 控制精准<br>✅ 性能高 | ❌ 会"忘记"早期对话 | 聊天机器人、客服场景 |
| **令牌缓冲记忆（Token-based）** | 保留不超过 max_tokens 的历史 | ✅ 适配模型上下文长<br>✅ 避免截断错误 | ❌ 仍会丢失远期信息 | 中长对话、成本敏感型应用 |
| **摘要总结记忆（Summary）** | 用 LLM 把历史压缩成摘要 | ✅ 支持超长对话<br>✅ Token 增长慢 | ❌ 细节丢失<br>❌ 需额外调用 LLM 生成摘要 | 会议纪要、长周期 Agent |
| **摘要+缓冲混合记忆** | 近期原始 + 远期摘要 | ✅ 短期精准 + 长期模糊记忆<br>✅ Token 效率高 | ❌ 实现复杂<br>❌ 依赖摘要质量 | 高级 Agent、个性化助手 |
| **实体记忆（Entity Memory）** | 追踪对话中出现的重要实体（人名、地点、偏好等）并持续更新 | ✅ 可精准跟踪用户信息<br>✅ 提升对话个性化和上下文理解 | ❌ 需要设计实体提取与更新逻辑<br>❌ 对实体识别错误敏感 | 个性化助手、长周期对话、CRM 系统 |
| **向量存储库记忆（Vector Store）** | 用 Embedding + 向量库检索相关历史 | ✅ 理论上无限记忆<br>✅ 保留细节 | ❌ 需要向量数据库<br>❌ 检索质量决定效果 | 知识库问答、企业级 RAG |

| 需求 | 推荐方案 |

| 要关系 | → 实体 |
