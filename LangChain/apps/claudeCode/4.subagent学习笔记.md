# LangChain Subagent 学习笔记

## 概述

本文通过分析 `4.subagent.ts` 文件，学习如何通过子代理实现上下文隔离。**核心洞察：进程隔离天然带来上下文隔离。**

## 核心架构

### 父子代理架构

```
Parent agent                     Subagent
+------------------+             +------------------+
| messages=[...]   |             | messages=[]      |  <-- fresh
|                  |  dispatch   |                  |
| tool: task       | ---------->| while tool_use:  |
|   prompt="..."   |            |   call tools     |
|   description="" |            |   append results |
|                  |  summary   |                  |
|   result = "..." | <--------- | return last text |
+------------------+             +------------------+
          |
Parent context stays clean.
Subagent context is discarded.
```

### 与 s03 的区别

| 方面 | s03 (todo) | s04 (subagent) |
|------|------------|----------------|
| 工具数量 | 5个 | **6个** (+task) |
| 上下文结构 | 单层 | **双层** |
| 工具集 | 统一 | **分层** |
| 上下文隔离 | 无 | **有** |
| 返回内容 | 全量 | **摘要** |

### 2. 分层工具集

```typescript
// 子代理工具：仅基础工具，不能递归生成子代理
const CHILD_TOOLS = [bashTool, readTool, writeTool, editTool];

// 父代理工具：基础工具 + task 工具
```

```
└── edit_file
```

### 3. Task 工具定义

```typescript
const taskTool = tool(
  (input: { prompt: string; description?: string }) => {
    // 占位符 - 实际执行在 agentLoop 中
    return input.prompt;
  },
  {
    name: "task",
    description:
      "Spawn a subagent with fresh context. It shares the filesystem but not conversation history.",
    schema: {
      type: "object",
      properties: {
        prompt: { type: "string" },
        description: {
          type: "string",
          description: "Short description of the task",
        },
      },
      required: ["prompt"],
    },
  }
);
```

**关键说明：**
- `prompt`: 传递给子代理的任务描述（必填）
- `description`: 任务的简短描述（可选，用于日志）
- 工具函数只是占位符，实际执行在父代理循环中

### 4. 子代理执行函数

```typescript
async function runSubagent(prompt: string): Promise<string> {
  // 1. 全新上下文
  const subMessages: BaseMessage[] = [
    new HumanMessage({ content: prompt }), // fresh context
  ];

  // 2. 安全限制：最多30轮
  for (let i = 0; i < 30; i++) {
    const messagesWithSystem = [
      new SystemMessage({ content: SUBAGENT_SYSTEM_PROMPT }),
      ...subMessages,
    ];

    const response = await childLlmWithTools.invoke(messagesWithSystem);
    subMessages.push(response);

    // 3. 检查是否完成
    if (response.response_metadata?.finish_reason !== "tool_calls") {
      break;  // 没有工具调用，任务完成
    }

    // 4. 执行工具调用
    if (response.tool_calls && response.tool_calls.length > 0) {
      const results: ToolMessage[] = [];

      for (const toolCall of response.tool_calls) {
        const handler = TOOL_HANDLERS[toolCall.name];
        let output: string;

        try {
          output = handler 
            ? handler(toolCall.args) 
            : `Unknown tool: ${toolCall.name}`;
        } catch (error: any) {
          output = `Error: ${error.message}`;
        }

        results.push(new ToolMessage({
          tool_call_id: toolCall.id!,
          content: output,
        }));
      }

      subMessages.push(...results);
    }
  }

  // 5. 只返回最终文本给父代理 -- 子代理上下文被丢弃
  const lastMessage = subMessages[subMessages.length - 1];
  if (lastMessage && typeof lastMessage.content === "string") {
    return lastMessage.content || "(no summary)";
  }
  return "(no summary)";
}
```

**执行流程：**
```
prompt
  │
  ▼
┌─────────────────────┐
│ 新建 subMessages=[] │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│ 添加 HumanMessage   │
│ (prompt)           │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐      ┌──────────────┐
│ 调用 childLLM       │◄────►│ 最多30轮循环  │
└──────────┬──────────┘      └──────────────┘
           │
           ▼
┌─────────────────────┐
│ 执行工具调用         │
│ (bash/read/write)   │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│ 直到 finish_reason  │
│ != "tool_calls"     │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│ 返回最终文本摘要     │
│ (丢弃上下文)         │
└─────────────────────┘
```

### 5. 父代理循环

```typescript
const agentLoop = async (messages: BaseMessage[]) => {
  while (true) {
    const messagesWithSystem = [
      new SystemMessage({ content: SYSTEM_PROMPT }),
      ...messages,
    ];

    const response = await parentLlmWithTools.invoke(messagesWithSystem);
    messages.push(response);

    if (response.response_metadata?.finish_reason !== "tool_calls") {
      return;
    }

    const results: ToolMessage[] = [];

    if (response.tool_calls) {
      for (const toolCall of response.tool_calls) {
        let output: string;

        if (toolCall.name === "task") {
          // 特殊处理：生成子代理
          const desc = (toolCall.args.description as string) || "subtask";
          const prompt = (toolCall.args.prompt as string) || "";
          console.log(`\n> task (${desc}): ${prompt.slice(0, 80)}`);
          output = await runSubagent(prompt);  // 等待子代理完成
        } else {
          // 常规工具调用
          const handler = TOOL_HANDLERS[toolCall.name];
          try {
            output = handler 
              ? handler(toolCall.args) 
              : `Unknown tool: ${toolCall.name}`;
          } catch (error: any) {
            output = `Error: ${error.message}`;
          }
        }

        console.log(`  ${output.slice(0, 200)}`);
        results.push(new ToolMessage({
          tool_call_id: toolCall.id!,
          content: output,
        }));
      }
    }

    messages.push(...results);
  }
};
```

**关键分支：**
```typescript
if (toolCall.name === "task") {
  // 子代理路径：等待执行完成，返回摘要
  output = await runSubagent(prompt);
} else {
  // 常规工具路径：立即执行，返回结果
  output = handler(toolCall.args);
}
```

## 核心设计模式

### 1. 上下文隔离模式

```
Parent Context (持久)
├── messages: [HumanMessage, AIMessage, ToolMessage, ...]
├── 保留完整对话历史
└── 可能变得很长

Child Context (临时)
├── messages: [HumanMessage(prompt)]  ← 只有任务描述
├── 执行完毕后丢弃
└── 永远保持干净
```

**好处：**
| 方面 | 效果 |
|------|------|
| Token 消耗 | 父上下文不会被污染 |
| 执行效率 | 避免长上下文推理 |
| 错误隔离 | 子代理错误不影响父代理 |
| 任务聚焦 | 子代理专注单一任务 |

### 2. 工具分层模式

```typescript
// 防止递归：子代理不能生成子代理
const CHILD_TOOLS = [bashTool, readTool, writeTool, editTool];
// 缺少 taskTool

const PARENT_TOOLS = [...CHILD_TOOLS, taskTool];
```

**设计考量：**
- 防止无限递归：子代理不能再生成子代理
- 权限控制：不同层级有不同的能力
- 资源管理：限制子代理的资源消耗

### 3. 摘要返回模式

```typescript
// 子代理只返回最终文本
const lastMessage = subMessages[subMessages.length - 1];
if (lastMessage && typeof lastMessage.content === "string") {
  return lastMessage.content || "(no summary)";
}
```

**信息流动：**
```
子代理完整执行过程（可能几百条消息）
           │
           ▼ 过滤
    最终摘要（几句话）
           │
           ▼ 返回
    父代理上下文（只增加一条 ToolMessage）
```

### 4. 安全限制模式

```typescript
for (let i = 0; i < 30; i++) {  // 安全限制
  // ... 子代理循环
}
```

**多层防护：**
| 防护层 | 机制 | 目的 |
|--------|------|------|
| 循环上限 | `i < 30` | 防止无限循环 |
| 工具黑名单 | bash 中的危险命令过滤 | 防止系统破坏 |
| 路径验证 | `safePath()` | 防止文件系统逃逸 |
| 工具分层 | 子代理无 task | 防止递归爆炸 |

## 执行流程示例

### 示例场景：分析项目依赖

```
用户: 分析项目的依赖情况并提出优化建议

父代理思考: 这是一个复杂任务，需要先了解项目结构，再分析依赖。
            我应该用 task 工具委派给子代理。

父代理调用: task(prompt="分析 package.json 中的依赖，检查版本冲突和安全漏洞")

子代理启动: messages=[HumanMessage("分析 package.json...")]
子代理执行: 
  - read_file("package.json") → 获取依赖列表
  - bash("npm outdated") → 检查过时依赖
  - bash("npm audit") → 检查安全漏洞
  - 分析并生成报告

子代理返回: "发现3个过时依赖和1个高危漏洞。lodash 可以升级到 4.17.21..."

父代理继续: messages=[..., ToolMessage("发现3个过时依赖...")]
父代理输出: "我分析了你的项目依赖，发现了以下问题：..."
```

### 消息流动详解

```
Round 1 (父代理):
  messages: [HumanMessage("分析项目的依赖情况")]
  response: AIMessage(content="", tool_calls=[{name: "task", args: {...}}])

Round 2 (父代理处理 task):
  检测到 tool.name === "task"
  → 调用 runSubagent(prompt)

Round 2.1-2.N (子代理):
  subMessages: [HumanMessage(prompt)]
  for i in 0..30:
    response = childLLM.invoke(subMessages)
    subMessages.push(response)
    
    if no tool_calls: break
    
    for tool in response.tool_calls:
      result = execute(tool)
      subMessages.push(ToolMessage(result))

Round 2.N+1 (子代理返回):
  return subMessages[-1].content  // 只返回最终文本

Round 3 (父代理继续):
  messages: [
    HumanMessage("分析项目的依赖情况"),
    AIMessage(content="", tool_calls=[...]),
    ToolMessage(content="发现3个过时依赖...")  // 子代理摘要
  ]
  response: AIMessage(content="我分析了你的项目依赖...")
```

## 关键学习点

### 1. 上下文管理策略

| 策略 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| 单层上下文 | 简单任务 | 简单直接 | 长任务会膨胀 |
| **双层上下文** | 复杂任务 | 上下文隔离 | 实现复杂 |
| 滑动窗口 | 超长对话 | 控制长度 | 可能丢失信息 |
| 摘要压缩 | 历史对话 | 保留关键信息 | 摘要可能不准确 |

### 2. 子代理的设计权衡

```typescript
// 权衡1: 循环上限
for (let i = 0; i < 30; i++)  // 为什么是30？
// 太少: 复杂任务无法完成
// 太多: 可能陷入死循环，消耗资源
// 30: 平衡点，覆盖大多数任务

// 权衡2: 工具集
const CHILD_TOOLS = [bash, read, write, edit]  // 为什么不给 task？
// 给 task: 理论上更强大，但有递归爆炸风险
// 不给 task: 安全，但子代理不能委派子任务
// 折中: 不给，子代理专注执行

// 权衡3: 返回内容
return subMessages[-1].content  // 为什么只返回最后一条？
// 返回全部: 父上下文膨胀，失去隔离意义
// 返回摘要: 保持干净，但可能丢失细节
// 折中: 返回摘要，关键信息由子代理总结
```

### 3. 错误传播策略

```typescript
// 子代理错误不传播到父代理
try {
  output = handler(toolCall.args);
} catch (error: any) {
  output = `Error: ${error.message}`;  // 错误转化为文本
}

// 最终被包装成 ToolMessage
results.push(new ToolMessage({
  tool_call_id: toolCall.id!,
  content: output,  // 可能包含 "Error: ..."
}));
```

**设计哲学：**
- 错误不中断流程
- 错误信息返回给 LLM
- 让模型决定如何处理错误

## 扩展方向

### 1. 多层嵌套

```typescript
// 允许子代理生成子代理
const CHILD_TOOLS = [...BASE_TOOLS, taskTool];  // 添加 task

// 添加深度限制
async function runSubagent(prompt: string, depth: number = 0) {
  if (depth >= 3) throw new Error("Max recursion depth reached");

  if (toolCall.name === "task") {
    output = await runSubagent(prompt, depth + 1);
```

```typescript
// 缓存子代理结果，避免重复计算
const subagentCache = new Map<string, string>();

async function runSubagent(prompt: string) {
  const cacheKey = hashPrompt(prompt);
  
  if (subagentCache.has(cacheKey)) {
    return subagentCache.get(cacheKey)!;
  }
  
  const result = await executeSubagent(prompt);
  subagentCache.set(cacheKey, result);
  return result;
}
```

## 总结

### 核心学习点

1. **上下文隔离**: 子代理使用全新上下文，执行完毕即丢弃
2. **工具分层**: 子代理工具集受限，防止无限递归
3. **摘要返回**: 只返回关键结果，保持父上下文干净
4. **安全限制**: 循环上限、工具黑名单、路径验证

### 设计原则

- **隔离原则**: 子代理上下文与父代理隔离
- **安全原则**: 多层防护防止失控
- **最小权限**: 子代理只获得必要工具
- **信息最小化**: 只返回必要信息

### 权衡决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 嵌套深度 | 1层 | 简单可控 |
| 工具集 | 基础4个 | 防止递归 |
| 循环上限 | 30轮 | 平衡效率和安全 |
| 返回内容 | 最终文本 | 保持父上下文干净 |
