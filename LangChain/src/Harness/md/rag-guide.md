# RAG 学习指南：理解、构建、优化

---

## 1. 什么是 RAG？

### 1.1 一句话定义

**RAG = Retrieval-Augmented Generation（检索增强生成）**

先从外部知识库检索相关内容，再把检索结果喂给 LLM，让它基于事实生成回答。

### 1.2 为什么需要 RAG？

LLM 单独使用时存在三个根本性问题：

| 问题 | 例子 | RAG 如何解决 |
|------|------|-------------|
| **知识时效性** | "昨天发布的新闻你知道吗？" → 不知道 | 检索最新文档，LLM 基于检索结果回答 |
| **私有数据盲区** | "我上周写的笔记在哪？" → 无法访问 | 索引你的本地文档，LLM 可以引用 |
| **幻觉** | 编造不存在的事实 | Prompt 中注入"只基于以下文档回答"，约束 LLM |

**类比理解**：LLM 就像一个博览群书但记忆模糊的专家。RAG 相当于在回答前先让他查一遍资料——回答质量立刻提升。

### 1.3 RAG vs 纯 LLM vs 微调

| 方案 | 成本 | 时效性 | 私有数据 | 适合场景 |
|------|------|--------|---------|---------|
| **纯 LLM** | 低（API 调用费） | ❌ 受训练截止日限制 | ❌ 无法访问 | 通用问答 |
| **RAG** | 中（检索+LLM） | ✅ 实时检索 | ✅ 索引私有文档 | 知识问答、文档搜索 |
| **微调** | 高（训练成本） | ❌ 仍是静态知识 | ⚠️ 需重新训练 | 特定风格/领域内化 |

**结论**：90% 的"让 LLM 了解我的业务"场景，用 RAG 就够了，不需要微调。

## 2. RAG 的完整流程

### 2.1 两阶段模型

RAG 分为两个完全不同的阶段：

```
┌────────────── 索引阶段（离线，一次性或定期）──────────────┐
│                                                            │
│  原始文档 → 分块(Chunking) → 嵌入(Embedding) → 存入向量库    │
│                                                            │
│  目标：把你的知识库建好索引，随时可以被检索                    │
└────────────────────────────────────────────────────────────┘

┌────────────── 查询阶段（在线，每次用户提问）──────────────┐
│                                                            │
│  用户提问 → 嵌入(Embedding) → 检索(Retrieval)              │
│  → 构造Prompt → LLM生成(Generation) → 返回回答              │
│                                                            │
│  目标：找到最相关的文档片段，让 LLM 基于事实回答               │
└────────────────────────────────────────────────────────────┘
```

**关键理解**：索引阶段是"建图书馆"，查询阶段是"去图书馆找书再写书评"。两者可以完全独立——你可以先建好索引，再反复查询。

### 2.2 五步详解

#### Step 1: Embedding（嵌入）— 文本变向量

**原理**：Embedding 模型把一段文字映射成一个数值向量（如 `[0.12, -0.34, 0.56, ...]`）。

**核心特性**：语义相似的文本 → 向量距离近。

```
"检索增强生成" → [0.8, 0.1, 0.6, ...]
"RAG 技术"     → [0.7, 0.2, 0.5, ...]   ← 两个向量很接近！

"足球比赛规则"  → [-0.3, 0.9, -0.2, ...] ← 和上面两个向量很远
```

**为什么要这样做？** 因为计算机无法直接比较"两段文字的含义是否相似"，但可以快速计算两个向量的距离（余弦相似度）。Embedding 把语义问题变成了数学问题。

**常用模型**：
- `nomic-embed-text`（Ollama 本地，768维，免费）
- `text-embedding-3-small`（OpenAI，1536维，按量计费）
- `BGE-large-zh`（开源，1024维，中文效果好）

#### Step 2: Chunking（分块）— 长文档切小块

**为什么要分块？**

1. **Embedding 有长度限制**：模型最大输入约 8K tokens，超了就截断
2. **向量"平均化"问题**：整篇文档 Embedding 后，向量是所有内容的"平均值"，丢失细节。一篇文档既讲 RAG 又讲微调，Embedding 后变成一个"模糊的AI技术向量"，检索时什么都匹配不上
3. **检索精度**：小块可以精准匹配到具体段落

**分块参数**：

| 参数 | 含义 | 经验值 |
|------|------|--------|
| `chunkSize` | 每块最大字符数 | 500-1000（太短语义不完整，太长精度差） |
| `chunkOverlap` | 相邻块重叠字符数 | chunkSize 的 10-20%（避免边界处的语义断裂） |

**分块策略**（按文档类型选择）：

| 文档类型 | 策略 | 原因 |
|----------|------|------|
| Markdown | 按 Header（# ## ###）拆分 | 保留结构，每个 section 是独立的语义单元 |
| 纯文本 | 递归字符拆分 | 通用方案，按段落→句子→词逐级切分 |
| 代码 | 按 AST（函数/类）拆分 | 函数级是最合理的语义单元 |

**最常见的坑**：chunkSize 设太大（>2000），导致检索粒度太粗，找到的是整篇文档而非具体段落。

#### Step 3: Indexing（索引）— 存入向量库

**做什么**：把分块后的每个小块 Embedding 成向量，连同原文和 metadata 存入向量数据库。

**向量数据库做什么**：存储 (向量, 原文, metadata) 三元组，支持高效的向量相似度搜索。

**向量数据库选择**：

| 数据库 | 特点 | 适合 |
|--------|------|------|
| **Chroma** | 轻量、本地、HTTP API | 开发/小规模 |
| **FAISS** | 纯计算库、最快 | 内存够、不需要持久化 |
| **Pinecone** | 云端、免运维 | 生产/不想管基础设施 |
| **Milvus** | 分布式、高性能 | 大规模生产 |

**metadata 的重要性**：存入向量库时，每个文档片段都要带上 metadata（来源、tags、时间等）。检索时可以用 metadata 做过滤（"只搜索 tags=work 的文档"）。

#### Step 4: Retrieval（检索）— 找到相关文档

**基本流程**：
1. 用户提问 → Embedding 成查询向量
2. 查询向量 vs 向量库中所有向量 → 按 cosine similarity 排序
3. 返回 top-K 个最相似的文档片段

**但单纯向量检索有弱点**：它擅长语义匹配但不擅长精确匹配。搜索"LangChain"可能匹配到"AI框架"而非精确包含"LangChain"这个词的文档。

**解决方案：混合检索**

| 检索方式 | 擅长 | 弱点 | 权重建议 |
|----------|------|------|---------|
| **向量检索** | 语义相似（"AI助手"≈"智能助理"） | 精确关键词匹配弱 | 0.7 |
| **关键词检索**（BM25/TF） | 精确字面匹配（"LangChain"精确命中） | 语义理解弱 | 0.3 |

加权融合：`finalScore = vectorScore × 0.7 + keywordScore × 0.3`

**检索的关键参数**：
- `topK`：返回多少条结果（通常 5-10）
- `threshold`：相似度阈值，低于此值的结果丢弃（通常 0.7）

#### Step 5: Generation（生成）— LLM 基于事实回答

**Prompt 构造**：把检索到的文档片段拼接为 context，注入到 Prompt 中：

```
你是一个知识助手。请仅基于以下文档内容回答用户的问题。
如果文档中没有相关信息，请明确说明"无法回答"，不要编造。

检索到的文档：
[文档1] (来源: note.md, 相关度: 0.92)
RAG 是检索增强生成技术...

[文档2] (来源: wiki.md, 相关度: 0.85)
Embedding 将文本转为向量...

用户问题：什么是 RAG？

请基于上述文档回答：
```

**Prompt 设计的关键原则**：
1. **明确约束**："只基于以下文档回答"——减少幻觉
2. **声明无知**："无法回答时请说明"——避免强行编造
3. **标注来源**：每个文档片段标注来源和相似度——让 LLM 知道哪些更可信

## 3. 写好 RAG 需要做的工作清单

### 3.1 必做项（没有这些 RAG 就跑不起来）

| # | 工作 | 具体内容 |
|---|------|---------|
| 1 | **选 Embedding 模型** | 确定维度、语言支持、本地/远程 |
| 2 | **选向量数据库** | Chroma/FAISS/Pinecone 等 |
| 3 | **确定分块策略和参数** | chunkSize、chunkOverlap、按文档类型选策略 |
| 4 | **实现索引流程** | 文档加载→分块→Embedding→入库 |
| 5 | **实现检索流程** | query→Embedding→向量搜索→topK |
| 6 | **构造 RAG Prompt** | context 拼接 + 约束指令 |
| 7 | **调用 LLM 生成回答** | 选 LLM、调 API |

### 3.2 应做项（提升 RAG 质量）

| # | 工作 | 效果 |
|---|------|------|
| 8 | **混合检索**（向量+关键词） | 解决向量检索的精确匹配弱点 |
| 9 | **metadata 过滤** | 检索时按 tags/source/date 过滤，减少噪音 |
| 10 | **阈值过滤** | 丢弃低相似度结果，避免喂给 LLM 噪音文档 |
| 11 | **去重** | 同一文档可能被多种方式检索到 |
| 12 | **分块策略适配** | Markdown 按 Header、代码按 AST，而非一刀切 |

### 3.3 高级优化（写好 RAG 的关键）

| # | 工作 | 效果 |
|---|------|------|
| 13 | **重排序 (Reranking)** | 初检索 top-20 → 用更强模型/规则重排 → 取 top-5 |
| 14 | **时间衰减** | 越新的文档权重越高（0.95^n 衰减公式） |
| 15 | **查询改写** | LLM 先把用户模糊提问改写为清晰检索 query |
| 16 | **多路召回** | 同一 query 生成多个变体，分别检索再合并 |
| 17 | **自检 (Self-RAG)** | LLM 生成后自我评估："我的回答是否有据可依？" |

## 4. 如何写好 RAG

### 4.1 分块是 RAG 质量的第一道门

分块质量直接决定检索精度。**分块做得好，RAG 已经成功了一半**。

**常见错误**：
- ❌ chunkSize 太大（>2000）→ 检索粒度粗，找到的是整篇文档
- ❌ chunkOverlap 太小（<50）→ 边界处信息丢失
- ❌ 所有文档用同一种策略 → Markdown/代码应区别对待
- ❌ 分块时不保留 metadata → 检索时无法按条件过滤

**最佳实践**：
- ✅ Markdown 按 Header 拆分，保留层级信息
- ✅ 代码按函数/类拆分，保留 import 和签名
- ✅ chunkSize 500-1000，chunkOverlap 10-20%
- ✅ 每个块都带 metadata（source, tags, created_at）

### 4.2 检索是 RAG 质量的第二道门

**检索质量 = LLM 看到什么内容 = 回答质量的上限**。

如果检索到的是噪音文档，LLM 再强也无法给出好回答。

**常见错误**：
- ❌ 只用向量检索 → 精确关键词匹配弱
- ❌ topK 太小（3）→ 可能漏掉关键文档
- ❌ topK 太大（20）→ 喂给 LLM 噪音太多，反而降低质量
- ❌ 不做阈值过滤 → 低质量结果混入 context

**最佳实践**：
- ✅ 混合检索（向量 0.7 + 关键词 0.3）
- ✅ topK 5-10
- ✅ threshold 0.7（丢弃低相似度结果）
- ✅ 去重（同一内容可能被两种方式检索到）

### 4.3 Prompt 是 RAG 质量的第三道门

**Prompt 决定 LLM 如何使用检索到的内容**。

**常见错误**：
- ❌ Prompt 不约束 LLM → LLM 可能忽略检索结果，自行编造
- ❌ 不标注来源 → LLM 无法判断哪些文档更可信
- ❌ context 太长（>4K tokens）→ LLM 注意力分散，关键信息被淹没

**最佳实践**：
- ✅ 明确约束："只基于以下文档回答"
- ✅ 标注来源和相似度
- ✅ context 控制在 2-3K tokens 以内
- ✅ 要求 LLM 在无法回答时明确说明

### 4.4 重排序是进阶的关键

初检索用 Embedding 做快速筛选（得到 top-20），再用更精确的方法重排（取 top-5）。

为什么需要重排序？因为 Embedding 模型的相似度计算是"粗筛"，可能有假阳性（看起来相似但实际不相关）。

**重排序方法**：
1. **Cross-Encoder 重排**：用一个专门的 reranker 模型（如 Cohere Rerank）对 (query, doc) 逐对打分，比 Embedding 的 Bi-Encoder 更精确
2. **规则重排**：时间衰减、标签匹配加分、来源优先级
3. **LLM 重排**：让 LLM 逐一评估每个文档的相关性（最精确但最贵）

### 4.5 查询改写是隐藏的杠杆

用户提问往往模糊、口语化，直接拿去检索效果差。

**例子**：
- 用户问："那个东西怎么用？" → 检索什么都匹配不上
- 改写后："LangChain RAG 模块的使用方法" → 精准检索

**改写方式**：
1. **LLM 改写**：先让 LLM 把用户提问改写为清晰检索 query
2. **多路生成**：一个提问生成 3-5 个不同角度的检索 query，分别检索再合并
3. **HyDE**：让 LLM 先假设一个答案，用假设答案去检索（答案和文档语义更接近）

## 5. RAG 效果评估

### 5.1 怎么知道 RAG 做得好不好？

| 指标 | 衡量 | 计算方式 |
|------|------|---------|
| **检索命中率** | 检索结果是否包含正确答案 | 有答案文档出现在 top-K 中的比例 |
| **检索 MRR** | 正确答案排第几位 | 1/排名位置 的平均值 |
| **回答忠实度** | LLM 回答是否基于检索文档 | 人工评估或用 LLM 自动评估 |
| **回答正确性** | 回答内容是否事实正确 | 人工评估或对照标准答案 |

### 5.2 常见 RAG 失败模式

| 失败模式 | 原因 | 解决方案 |
|----------|------|---------|
| **检索不到** | 分块太粗 / Embedding 模型不适合 / query 模糊 | 优化分块、换模型、查询改写 |
| **检索到噪音** | topK 太大 / 无阈值过滤 / 无 metadata 过滤 | 混合检索+阈值+过滤+重排序 |
| **LLM 忽略检索结果** | Prompt 未约束 / context 太长 | 明确约束 + 控制 context 长度 |
| **LLM 幻觉** | 检索结果不包含答案 / Prompt 没要求声明无知 | 自检机制 + 声明无知指令 |

## 6. 从零开始写 RAG 的推荐顺序

```
Step 1: 先跑通最小 RAG
        ──────────────────
        手写 5 个文档片段 → 直接 Embedding → Chroma 存储
        → 简单向量检索 → Prompt + LLM → 得到回答
        目标：理解流程，看到效果

Step 2: 加入分块
        ──────────────────
        用真实长文档 → 递归字符分块 → Embedding → Chroma
        → 检索 → 生成
        目标：理解分块参数对检索质量的影响

Step 3: 加入混合检索
        ──────────────────
        向量检索 + 关键词检索 → 加权融合 → 去重 → 阈值过滤
        目标：对比纯向量 vs 混合检索的质量差异

Step 4: 优化分块策略
        ──────────────────
        Markdown 按 Header、代码按 AST 分块
        目标：不同文档类型用不同策略

Step 5: 加入重排序和查询改写
        ──────────────────
        初检索 top-20 → 重排取 top-5
        用户提问 → LLM 改写 → 多路检索 → 合并
        目标：达到生产级 RAG 质量
```

**每一步都要对比评估**：改了什么 → 检索命中率变了多少 → 回答质量变了多少。不评估就无法知道优化是否有效。
